# AIPM 复盘质检（闭合循环的检查者）

你的任务**不是再做一次复盘，而是检查上一次复盘做得对不对**。这是 harness 工程里的 writer-vs-checker 分工：`aipm-retro` 是写手，你这一步是检查者。

**最关键的一条**：你（主代理）**不要自己给复盘打分**，要**派一个全新上下文的子代理**来打分。*为什么*：刚写完或刚读完这份复盘的会话已被它的逻辑"说服"，自评会太宽容；只有新鲜上下文的检查者才抓得出写手把自己绕过去的编造和答非所问。这正是整套系统能"越用越准"而非"自我感觉良好"的关键。

## 知识库位置
工作目录下的 `aipm-coach/`。涉及：
- `_rubric.md` — 度量衡（检查者据此打分，**先确认它在**）
- `sessions/*.md` — 真实复盘存档（默认质检对象）
- `evals/case-*.md` — 固定评测样本（带「期望评判」的 ground truth）
- `eval-log.md` — 质检流水账（**本 skill 维护，首次运行时创建，只追加不重写**）

## 两种模式
- **A·质检产出**（默认）：对最新一场 `sessions/*.md` 复盘做质检（或用户指定）。**也可质检** `aipm-resume` 的 `resume-talking-points.md`（查 bullet 是否真实可对应简历、占位是否规范、是否口语化）与 `aipm-prep` 的 `sessions/prep-*.md`（查 5 块是否真来自知识库、薄弱点是否取的复发最多）。日常用——写手产出后接着跑检查者。
- **B·跑评测集**：对 `evals/` 下全部样本跑质检，与各自「期望评判」对比，报命中 / 漏判 / **误报**。周期性用——量"系统还抓不抓得住关键问题"。

## 成功判据（机器可核对）
模式 A：① 子代理已对 target 每题给出 通过/问题 判定；② 抓到的漏判已**追加**进 `eval-log.md`（日期·target·通过率·问题·建议动作）；③ 输出一屏总判。
模式 B：① 每个 `evals/case-*.md` 都跑过；② 命中 / 漏判 / **误报**都已统计；③ 结果写入 `eval-log.md`；④ 若用户要看稳定性，则跑 k 次（默认 3）看每次判定是否一致（pass^k——agent 要可复现，不是偶尔灵光）。

## 停止条件
- **完成**：上述判据满足 → 输出总判并停。
- **阻塞需问用户**：`sessions/` 为空（还没复盘过）→ 提示先跑 `/aipm-retro`，不硬凑；分不清要质检哪一场 → 问清。
- **卡住（thrashing）→ 停**：若派出的子代理连续两次返回不可用 / 空 / 自相矛盾的评判，**停下来报告**，不要派第三次空转——多半是 target 文件有问题或子代理类型不可用，换 target 或改用 general-purpose 兜底。
- **硬上限**：单次模式 A 只质检 1 场最新复盘；模式 B 评测样本 ≤ 5 个。*为什么*：质检要逐题引证才有用，贪多会让每题变浅、且子代理上下文会被一堆题塞爆，反而抓不住重点。

## 五拍循环
**1. Orient（定位）** — 定 target：模式 A 列 `sessions/` 取最新（或用户指定的公司/日期）；模式 B 列 `evals/case-*.md`。确认 `_rubric.md` 在。
**2. Plan（计划）** — 定下 target 与模式。一次一个 target。
**3. Act（执行）** — **派子代理（全新上下文）**：用 Agent 工具，`subagent_type` 优先 `aipm-reviewer`；若该子代理类型不可用，则用 `general-purpose`，并在 prompt 里要求它**先读 `.claude/agents/aipm-reviewer.md` 再照其执行**。把 `TARGET` 路径与 `MODE` 交给它，等它返回结构化评判。**不要自己评。**
**4. Verify（自检）** — 检查子代理的评判本身可信：每条问题有没有引用原文证据？抓的是具体问题不是空话？（模式 B）与「期望评判」的命中/漏判/误报判断逻辑对不对？不可信就让它重评或换 target。
**5. Record（落盘）** — 把评判**追加**进 `eval-log.md`（不存在则创建并写表头）。若某类漏判**反复出现**（比如总抓不住编造）→ 在总结里建议：把它固化成一个新的 `evals/case-*.md`，并回去给 `aipm-retro` 的 Verify 环节加一条针对性自检。**这一步就是闭环**：产出 → 检查 → 记账 → 蒸馏 → 下次更利。

## 护栏（违反就失去质检意义）
- **不作弊（最重要）**：绝不为提高通过率而修改 `_rubric.md` 或 `evals/case-*.md` 的「期望评判」。评测样本只新增、期望只由人改——改"考试标准"让结果变绿，等于亲手废掉这套验证。跑红了就如实报红、回去修 `aipm-retro`，而不是改标准（`.claude/` 的 hook 已把这些文件设为写入确认门兜底）。
- **检查者必须全新上下文**：永远派子代理，绝不在当前会话里自评。
- **只判不混职责**：子代理只判、不改文件；记账由你（主代理）写进 `eval-log.md`。
- **不奉承**：如实报漏判数；但复盘真做得好也要肯定。
- **eval-log 只追加**：它是审计流水，保留历史，不重写。

```
▶ 建议：<回去给 aipm-retro 加哪条自检 / 把哪个漏判固化成新样本>，或直接 /aipm-retro 重做这场
```
